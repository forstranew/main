문제 31: Tomcat에 관한 설명으로 정확한 것은?
① Java 프로그램의 개발과 실행을 위한 플랫폼
② 80% 정도가 C 언어로 구현된 웹 서버
③ 아파치에서 개발한 서블릿/JSP 컨테이너
④ JSP 개발에 특화된 오픈 소스의 통합 개발 환경
Apache Tomcat은 아파치 소프트웨어 재단에서 개발한 서블릿/JSP 컨테이너입니다.
각 선택지 설명:
- ① Java 플랫폼: JDK (Java Development Kit)
- ② C 언어 웹 서버: Apache HTTP Server
- ③ 서블릿/JSP 컨테이너: Tomcat ✓
- ④ 통합 개발 환경: Eclipse, IntelliJ IDEA
문제 32: 이클립스를 이용하여 웹 애플리케이션 Example을 개발하였고 이것을 war 파일로 배포하려고 한다. 배포할 위치로서 아래 빈 부분에 들어갈 폴더 이름은 무엇인가?
[톰캣설치폴더]\_____\Example.war
① META-INF
② WEB-INF
③ WebContent
④ webapps
Tomcat에서 웹 애플리케이션을 배포할 때는 webapps 폴더에 WAR 파일을 저장합니다.
예: [톰캣설치폴더]\webapps\Example.war
- webapps: 웹 애플리케이션 배포 디렉터리
- META-INF: WAR 파일 내부 메타데이터 디렉터리
- WEB-INF: 웹 애플리케이션 설정 파일 디렉터리
- WebContent: Eclipse 프로젝트의 웹 리소스 폴더
문제 33: HTTP 요청 방식 중 GET 방식에 관한 설명은 무엇인가?
① URL에 쿼리 문자열을 붙여 데이터를 전달한다.
② 게시글과 같이 많은 양의 데이터를 보낼 때 적당하다.
③ 암호와 같은 민감한 정보를 보낼 때 적당하다.
④ HTTP 요청 메시지의 몸체에 데이터를 포함시켜 전달한다.
GET 방식은 URL에 쿼리 문자열을 붙여 데이터를 전달합니다.
예: http://example.com/page.jsp?name=Kim&age=20
GET vs POST 비교:
- GET: URL 노출, 길이 제한, 즐겨찾기 가능, 민감정보 부적합
- POST: 요청 본문에 데이터, 길이 제한 없음, URL 노출 안 됨, 민감정보 전송 적합
문제 34: JSP 페이지에서 사용할 클래스를 가져오기(import) 위한 선언으로 올바른 것은?
① <% import java.util.Date; %>
② <%@ import file="java.util.Date" %>
③ <%@ page import="java.util.Date" %>
④ <%@ page import="java.utilDate" %>
JSP에서 자바 클래스를 import하려면 page 지시어와 import 속성을 사용합니다.
예: <%@ page import="java.util.Date" %>

문제 36: JSP의 내장 객체인 response 객체가 제공하지 않는 작업은 무엇인가?
① getWriter() - 출력 스트림 객체를 리턴한다.
② getCookies() - 클라이언트가 보낸 쿠키 정보를 읽는다.
③ sendRedirect() - 요청을 다른 페이지로 이동시킨다.
④ setHeader() - 응답 헤더를 설정한다.
getCookies()는 request 객체의 메서드입니다. response 객체는 getWriter(), sendRedirect(), setHeader(), addCookie(), setContentType() 등을 제공합니다.

문제 39: request.setCharacterEncoding()의 의미는?
① 페이지를 파일로 저장할 때 사용할 charset을 지정
② request 영역에서 공유할 charset 속성을 지정
③ 요청 파라미터를 읽기 전에, 디코딩에 사용할 charset을 지정
④ 응답 메시지를 만들 때 사용할 charset을 지정
request.setCharacterEncoding()은 클라이언트가 전송한 요청 파라미터를 읽을 때 사용할 문자 인코딩을 지정하며, 반드시 getParameter() 호출 전에 설정해야 합니다.
문제 40: process.jsp에서 아이디를 출력하기 위해 빈칸 ㄷ에 들어갈 내용은?
① <% request.getParameter("id"); %>
② <% request.getParameter("name"); %>
③ <%= request.getParameter("id") %>
④ <%= request.getParameter("name") %>
값을 출력하려면 표현식 <%= %>을 사용하고, 파라미터 이름은 form에서 지정한 "id"를 사용해야 합니다.
문제 41: from.jsp에서 사용되지 않은 JSP 요소는 무엇인가?
① 스크립트릿
② 액션 태그
③ 지시어
④ 표현식
from.jsp에서는 지시어, 스크립트릿, 액션 태그는 사용되었으나 표현식은 사용되지 않았습니다. (to.jsp에서는 있음)
문제 42: request 영역에 속성을 저장할 때 빈칸 ㄱ에 들어갈 메서드는?
① getParameter
② setParameter
③ getAttribute
④ setAttribute
request.setAttribute()는 request 영역에 속성을 저장하여 forward/include 시 데이터 공유에 사용됩니다.
문제 43: 클라이언트가 from.jsp를 요청할 때 브라우저에서 보이는 결과는?
① URL: from.jsp, 화면: 여기는 to.jsp 페이지 a1
② URL: to.jsp, 화면: 여기는 to.jsp 페이지 p1
③ URL: from.jsp, 화면: from.jsp 페이지 - 포워딩 전, to.jsp 페이지 a1, from.jsp 페이지 - 포워딩 후
④ URL: from.jsp, 화면: from.jsp 페이지 - 포워딩 전, to.jsp 페이지 p1, from.jsp 페이지 - 포워딩 후
<jsp:forward>는 현재 페이지 실행을 즉시 중단하고 to.jsp로 제어권을 넘깁니다. forward 이후의 코드는 실행되지 않습니다.
문제 44: from.jsp에서 사용된 <jsp:param> 태그 설명 중 잘못된 것은?
① <jsp:param>은 <jsp:include>와 <jsp:forward>의 서브 요소로 사용된다.
② 요청을 다른 페이지로 포워딩하거나 할 때, 원래 요청에 파라미터를 추가하기 위해 사용된다.
③ 기존에 같은 이름의 파라미터가 존재하면 <jsp:param>에 지정된 값이 우선된다.
④ name과 value 속성은 생략 가능하다.
<jsp:param>의 name과 value 속성은 필수이며 생략할 수 없습니다.
문제 45: from.jsp와 to.jsp 페이지 사이에서 공유될 수 없는 JSP 내장 객체는?
① pageContext
② request
③ session
④ application
pageContext는 현재 페이지에서만 유효하며, 다른 페이지와 공유되지 않습니다. request, session, application은 공유 가능합니다.
